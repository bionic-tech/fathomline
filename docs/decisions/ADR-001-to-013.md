# ADR-001: Standalone product identity

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Fathom could be a module inside the owner's existing network-observability platform (sharing
repo, Authentik, PostgreSQL) or its own product. It shares that platform's *engineering
standards* (the Master Framework v0.7) but has a distinct domain (storage estate, not network
observability) and a possible open-source future.

## Decision
Build Fathom as a **standalone product** with its own repository, auth, and database,
**reusing** the Master Framework's SDLC, templates, and adversarial-review process.

## Consequences
### Positive
- Clean licensing/IP story for a possible public release (ADR-009/010 reinforce this).
- No coupling of the sibling platform's release cadence or schema to Fathom's.
### Negative
- Some duplication of plumbing (auth wiring, CI) that a module would have inherited.
### Risks
- Two codebases to keep standards-aligned; mitigated by sharing the framework verbatim.

# ADR-002: Agent-push over mTLS as the v1 transport

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
The estate is all Linux/TrueNAS for v1 (SMB/SFTP/Windows deferred to v1.2–1.5). Options:
agent-on-host pushing results, agentless SSH-exec, or central pull over NFS/SMB. Content
hashing over the network is prohibitively slow and gives the core filesystem access it
shouldn't have.

## Decision
**Agent-push over mTLS** is the sole v1 transport. Each host runs a collector that reads
locally and pushes deltas to the core; the core never mounts remote filesystems. SSH-exec
and SMB/SFTP connectors are designed-for (via the transport plugin boundary) but built in
v1.2–1.5.

## Consequences
### Positive
- Fastest path; no byte-hashing across the wire; core has zero remote FS access.
- mTLS gives per-host machine identity and instant revocation.
### Negative
- Requires deploying an agent to every host (acceptable — all hosts are ours).
### Risks
- Agent fleet management overhead; mitigated by ADR-013 (container + binary) and a cert
  fingerprint allow-list.

# ADR-003: PostgreSQL (Patroni) catalogue + SQLite staging; core on the NAS host

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
The estate is 50M+ files / 150TB+. The catalogue must scale and be HA; agents must not
lose work on a network blip. API service placement must sit on a stable app host.

## Decision
Central catalogue + snapshots on the **existing Patroni PostgreSQL cluster**;
each agent stages to a local **SQLite (WAL)** queue and pushes resumable deltas. The
**FastAPI core services run as containers on the NAS host**.

## Consequences
### Positive
- Reuses HA Postgres; partitioning + materialised paths handle 50M+ rows.
- Local staging makes pushes resumable; no scan lost to a transient outage.
### Negative
- The NAS host is both core host and a heavy scan target — must be load-aware.
### Risks
- Catalogue scale; mitigated by partitioning by `(host, volume)` and snapshot retention
  controls (ADR per data-model deep-dive).

# ADR-004: `StorageBackend` Protocol for filesystem extensibility

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Filesystems differ in how size, change-detection, and special features (compression,
reflinks, ADS) work: ZFS, ext4/XFS, btrfs, NTFS/exFAT (later). Hard-coding any one fails.

## Decision
A `StorageBackend` **Protocol** (structural typing) abstracts walk/stat/hash/usage. ZFS and
generic-POSIX ship first; others are plugins implementing the same interface.

## Consequences
### Positive
- New FS = a plugin + conformance tests, no core changes.
### Negative
- An abstraction layer to maintain.
### Risks
- Backend drift from FS truth; mitigated by per-backend fixture tests (symlinks, sparse,
  compressed, reflinks).

# ADR-005: Free chart stack (ECharts/D3/uPlot), not Highcharts

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Highcharts is not free for commercial/internal-business use. Fathom is "build as if we will
open-source" (ADR-001 context), so a redistributable chart stack is required.

## Decision
Use **Apache ECharts** (treemap, sunburst, pie, bar), **D3** (bespoke navigable treemap),
**uPlot** (dense time-series). Chart layer isolated behind one adapter so it's swappable.

## Consequences
### Positive
- No licensing exposure; full chart-type coverage; redistributable.
### Negative
- Slightly more integration effort than a single commercial suite.
### Risks
- None material. A Highcharts swap remains possible behind the adapter if ever licensed.

# ADR-006: Incremental change feed: `zfs diff` / `fanotify` / re-stat

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
At 50M+ files, repeated full walks are unacceptable. The first scan is heavy (warned);
after that, updates must be light-touch and event-driven.

## Decision
After the initial index: **`zfs diff`** between snapshots on ZFS, **`fanotify`** mount-wide
on other Linux FS, periodic **re-stat by mtime** as the fallback. Full re-baseline is an
explicit, warned operation.

## Consequences
### Positive
- Cheapest possible incrementals; ZFS already knows what changed.
### Negative
- Three code paths to maintain.
### Risks
- `fanotify` privilege requirements; resolved via the scoped reader user + capabilities.

# ADR-007: Stack — FastAPI + Pydantic v2 + async Python 3.12 + React

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
The platform needs an async API, strong input validation, and a modern UI, and should
inherit the existing code-quality/async/testing standards.

## Decision
**FastAPI + Pydantic v2 + async Python 3.12** backend, **React** frontend, Mermaid for docs
— identical to the framework's reference stack so CI gates and skills apply unchanged.

## Consequences
### Positive
- Reuses standards, tooling, and reviewer familiarity; great async I/O fit for scanning.
### Negative
- Python hashing throughput < native; mitigated by progressive hashing + the agent binary
  doing hot loops efficiently (BLAKE3), and ADR-013's static binary option.
### Risks
- None material.

# ADR-008: API-first `PlatformAdapter` control plane

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Host facts (pools, disks, RAID/resync state, usage) are best read from a NAS's authoritative
API rather than parsing CLI output. TrueNAS exposes a JSON-RPC 2.0 WebSocket API (REST
removed in TrueNAS 26); per-call `midclt` carries heavy auth/audit overhead.

## Decision
A read-only **`PlatformAdapter`** abstraction, **API-first with CLI fallback**. The TrueNAS
adapter holds **one persistent authenticated WebSocket session** (local socket on-box, API
key remote), never per-call `midclt`. Adapters are pluggable and community-extensible, and
**never on the write path**.

## Consequences
### Positive
- Authoritative topology/usage; resync guard reads `pool.status`; open-source extensible.
### Negative
- Per-vendor adapters to write (only TrueNAS + generic in v1).
### Risks
- Third-party adapter as an attack vector; mitigated by read-only-only contract + conformance
  + supply-chain checks (see STRIDE E-5).

# ADR-009: OIDC-first + local auth, pluggable IdP

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Authentik is the estate's SSO, but hard-coupling to it would block anyone cloning a public
repo who doesn't run Authentik.

## Decision
**OIDC-first** auth (Authentik as the reference IdP) **plus a local-auth option**, behind a
pluggable auth interface. Roles: viewer / operator / remediator / admin (deny-by-default).

## Consequences
### Positive
- Works with any OIDC IdP or standalone; open-source friendly.
### Negative
- Two auth paths to test (OIDC + local).
### Risks
- Local-auth misconfiguration; mitigated by MFA enforcement on privileged roles and the
  RBAC test matrix.

# ADR-010: Pluggable secret backend — Docker secrets + OpenBao

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Fathom holds mTLS keys, TrueNAS API keys, and later SMB/SSH creds. Secrets must never live
in code/config (Framework Principle #8). The backend should suit both simple and production
deployments and stay open-source friendly.

## Decision
**Pluggable secret backend** with **Docker secrets** (simple) and **OpenBao** (production)
as the v1 options, with a documented rotation schedule. No secrets in code/`.env`/config.

## Consequences
### Positive
- OSS-aligned (OpenBao); count-only credential logging.
### Negative
- Two backends to support.
### Risks
- Rotation drift; mitigated by the retention/rotation schedule (control-framework doc 24).

# ADR-011: Remediation enabled in v1 — human-selected, guarded

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Remediation (delete/hardlink/quarantine of duplicates) is the only data-destroying capability
and the highest-risk surface. The project owner wants it **live in v1**, not deferred.

## Decision
Remediation is **built and enabled in v1** under strict controls:
- **Human selection only.** Fathom never auto-selects which duplicate to remove. The dedup
  report presents each group; the operator explicitly selects keep/remove. A **non-binding,
  override-able suggested keeper** (configurable rules: oldest / preferred path or volume,
  reason shown) may be highlighted to speed selection, but commits nothing.
- **Per-job action choice:** quarantine-then-purge, hardlink-dedupe, or hard-delete.
- **Guards:** mandatory dry-run re-verification (inode/size/hash drift → abort);
  `openat`+`O_NOFOLLOW` against TOCTOU; signed single-use action jobs; **blast-radius caps
  with explicit confirmation**; **7-day quarantine** before purge-eligibility; audit-before-act.
- **Separation of duties:** the reader process/user cannot write; only `strata-actor` can,
  and only on approved paths.
- No two-person rule (single-operator v1; multi-user-capable later).

## Consequences
### Positive
- Hands-on cleanup in v1; no destructive automation possible.
### Negative
- Highest review burden — the write path must clear **zero open P0/P1** before release.
### Risks
- TOCTOU / wrong-file deletion / forged jobs; mitigated as above and tracked as STRIDE
  T-2/T-3, E-1; each ≥P2 finding gets a named regression test.

# ADR-012: CI on self-hosted Gitea/Forgejo runners

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
Mandatory CI gates (SAST, dependency scan, IaC scan, zero critical/high CVEs) must run
somewhere; the estate is local-first/cloud-free.

## Decision
CI on **self-hosted Gitea/Forgejo** with local runners; SAST + `pip-audit` + IaC scan as
required gates; signed releases; non-root containers mandatory.

## Consequences
### Positive
- Keeps the toolchain local; no cloud dependency; supports private or public mirror.
### Negative
- Runner maintenance is on us.
### Risks
- CI/CD tampering (supply chain); mitigated by signed releases + pinned deps + four-eyes on
  release (STRIDE E-6).

# ADR-013: Agent ships as both container and static binary

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
TrueNAS favours a container (Custom App); Ubuntu nodes can run either; SSH-exec (future)
wants a dependency-free binary.

## Decision
Ship the agent as **both** a container image and a **static binary**, from one codebase.

## Consequences
### Positive
- Each host uses the best fit; static binary enables future agentless SSH-exec.
### Negative
- Two release artefacts to build/sign.
### Risks
- Drift between artefacts; mitigated by building both in one signed CI pipeline.

# ADR-014: Sandboxed on-demand preview rendering

**Status:** Accepted **Date:** 2026-06-04 **Deciders:** project owner

## Context
v1 needs a file-manager-style **preview pane** (images, PDF, Office, text/code) so the
operator can decide what to keep, not just see sizes. This **reverses** Fathom's original
"metadata-only, never serve content" posture and reintroduces decoding of untrusted file
content — historically one of the richest sources of exploitable CVEs (PDF/Office/image
decoders).

## Decision
Render previews in a **central, sandboxed, egress-less, ephemeral preview worker**,
**on-demand** when a file is selected. Render **server-side to safe derived artifacts only**:
image thumbnail; **first-page(s) raster + extracted text snippet** for PDF/Office (LibreOffice
in the sandbox); safe syntax-highlighted render for text/code/markdown. **Only the derived
artifact reaches the browser** — never the raw original; never raw SVG/HTML. **Bounded LRU
cache, encrypted at rest, 30-minute TTL.** Role- and scope-gated. v1 types: images, PDF,
Office, text/code/markdown; **video/audio (poster/cover art via ffmpeg) deferred**.

## Consequences
### Positive
- Genuinely useful triage; any decoder exploit is contained in the sandbox, away from data
  and core; no raw content to the browser eliminates client-side render exploits.
### Negative
- New render machinery + a hardened sandbox to build and maintain; transient single-file
  byte transfer on click; render CPU cost.
### Risks
- Renderer/decoder CVEs, sandbox escape, decompression/page bombs, preview-cache-at-rest —
  mitigated by sandbox isolation (seccomp/gVisor), no egress, resource/time/page caps,
  non-root, minimal mounts, encrypted bounded cache. Tracked as STRIDE **T-6, I-7, I-8,
  D-6, E-7**; each ≥ P2 gets a named regression test.

## Implementation status (2026-06-14)
Both topologies are built and tested:
- **Single-host** (`preview_local_fetch=true`): the core provisions the runtime at startup with
  `LocalFileFetcher` (reads the file off local disk) + the gVisor `RunscSandboxDriver`.
- **Distributed** (the fleet shape: core on TrueNAS, gVisor worker on a separate host — AR-0002):
  the core hosts the `/preview` route + DB + grant minting and pulls each file from its owning
  agent over the **signed single-file pull** (`PreviewPullQueue` + `GrantPullFetcher` → the agent's
  read-only `preview-serve` daemon verifies a nonce'd, host-scoped, TTL-bounded, Ed25519-signed
  `FileGrant` and serves exactly the named inode), then ships the bytes to the worker's `/render`
  (`RemoteSandboxDriver` → the real `RunscSandboxDriver`, E-7 fail-closed). Enablement config + the
  review checklist live in `deploy/worker-1/README.md`.

**Integrity anchors (adversarial-review hardening).** The read != return boundary is enforced by:
(1) the served bytes' BLAKE3 is verified against the catalogue's `full_hash` when present (catches
an in-place rewrite that preserves the inode, and guarantees the encrypted cache key names the
served bytes); (2) `O_NOFOLLOW` + signed-64-normalised inode anchor, with a missing (`0`) inode
failing closed; (3) the agent serve body is capped at `preview_max_input_bytes` *before* decode
(no single-agent OOM); (4) remote-backend (SMB/SFTP/rclone) entries are refused for preview (their
synthetic path is not locally openable); (5) the grant queue drains expired grants (bounded
growth) and the daemon survives a malformed response. The new agent file-read surface remains
**review-gated for fleet enablement** until its dedicated adversarial-review round signs off.

## Implementation status addendum (2026-06-15) — LIVE, worker on node-2
The adversarial-review round + its P1/P2/P3 fixes are done and the owner greenlit fleet enablement,
so distributed preview is now **LIVE**. Operational notes from first deployment:
- **Worker host: `node-2` (203.0.113.241), not worker-1.** worker-1's docker data-root disk did not
  return after a reboot (its images were lost with it), so the worker was relocated. The placement
  requirement is unchanged — any **non-TrueNAS host with gVisor/`runsc` installed** works; `nas-1`
  (the core) routes to every fleet host, so the core→worker hop is host-agnostic.
- **Worker image = core image + the docker CLI** (`deploy/worker-1/Dockerfile.worker`). The runsc
  driver shells `docker run --runtime=runsc`, which the stock core image lacks; the per-render
  decoders live only in the separate `fathom-preview:local` sandbox image (`Dockerfile.preview`).
- **Real bug fixed (commit 9f972ae):** the sandbox driver double-specified the image ENTRYPOINT
  (`python -m fathom.preview.sandbox_entry`) → every render `rc=2` "unrecognized arguments". The
  driver now appends only the per-render arg flags. This path had never been exercised end-to-end.
- **core→worker transport is plain HTTP over the LAN** (auth by the shared `X-Fathom-Proxy-Secret`):
  the core's render client uses default-TLS httpx, which would reject a homelab cert, so HTTPS is not
  used on this hop — acceptable on the trusted private LAN; revisit if the hop ever leaves it.
- **Operational gotcha:** recreating `fathom-api` gives it a new docker IP, but nginx (`proxy`)
  caches the old upstream → `502` on ingest AND preview poll until the proxy is recreated. Always
  `up -d --force-recreate proxy` after an api recreate (follow-up: a docker `resolver` in the proxy).
- Enablement is a compose **override** (`deploy/nas-1/docker-compose.preview.yml`, default-OFF base)
  + the Ed25519 grant keypair (host-only). Verified end-to-end: authenticated `/preview/<id>` →
  grant → agent serve → node-2 gVisor render → derived artifacts (200 on .md/text/code).
