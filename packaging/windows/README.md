## What it produces

`dist/fathomline-agent.exe` â€” a self-contained x64 console executable (no Python install needed on
the target). It freezes `src/fathom/agent/__main__.py`, i.e. the same code reached by
`python -m fathom.agent`. Invocation is unchanged:

- `fathomline-agent.exe scan` â€” one `scan â†’ stage â†’ push` pass (the W1 default).
- `fathomline-agent.exe listen` â€” the ADR-025 signed-job listener (not a W1 deliverable, but the
  same entry module handles it, so it is frozen in for parity).

All wiring is read from the **environment**, never argv (no secrets on the command line):
`FATHOM_AGENT_CONFIG` (required â€” path to `agent.config.yaml`), `FATHOM_AGENT_STAGING`,
`FATHOM_AGENT_OPERATOR`, `FATHOM_AGENT_MODE`. With no `FATHOM_AGENT_CONFIG` the process logs an
error and exits `2`; a failed scan scope exits `1` â€” so the Scheduled Task / supervisor gets a real
signal.

## How to build

### On a Windows host

```powershell
# From anywhere; the script resolves paths relative to itself.
cd packaging\windows
# First build (or after changing deps / the spec): force a clean venv + cache.
.\build-agent-exe.ps1 -Clean
# Subsequent builds (idempotent, reuses the venv):
.\build-agent-exe.ps1
```

Output: `packaging\windows\dist\fathomline-agent.exe`, with its SHA256 printed at the end.

Requirements on the build host: Python 3.12+ (matches `requires-python` in `pyproject.toml`) with
the `py` launcher, or pass `-Python C:\path\to\python.exe`.

### In CI (`windows-latest`)

A `windows-latest` job runs the same script. Sketch:

```yaml
build-windows-agent-exe:
  runs-on: windows-latest
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with:
        python-version: "3.12"
    - name: Build frozen agent exe
      shell: pwsh
      run: ./packaging/windows/build-agent-exe.ps1 -Clean -Python python
    - uses: actions/upload-artifact@v4
      with:
        name: fathomline-agent-exe
        path: packaging/windows/dist/fathomline-agent.exe
```

(ADR-027 already calls for a `windows-latest` CI lane for the Windows path/walk tests; the exe build
is a natural addition to it. Wiring it into `.github/workflows/ci.yml` is a separate change â€” this
scaffolding does not modify CI.)

## How it slots into the enrolment flow

The Windows enrolment bundle is generated by `src/fathom/core/deploy/winbundle.py` (ADR-027 W1): a
**zip** containing `agent.config.yaml`, `certs\`, `run-scan.ps1`, `install-agent.ps1`, `README.txt`.
The installer registers a daily **Scheduled Task** that runs `run-scan.ps1` as `SYSTEM`.

`run-scan.ps1` already **prefers the frozen exe when present**, falling back to Python otherwise:

```powershell
$exe = Join-Path $Root 'fathomline-agent.exe'
if (Test-Path $exe) { & $exe scan } else { & py -3 -m fathom.agent scan }
```

So `fathomline-agent.exe` is a **drop-in replacement** for `py -3 -m fathom.agent`: place the built
exe next to `agent.config.yaml` in the bundle (same directory as `run-scan.ps1`, which sets
`$env:FATHOM_AGENT_CONFIG` / `$env:FATHOM_AGENT_STAGING` before launching it). No change to
`winbundle.py` is needed â€” the launcher contract already accounts for the exe. With the exe shipped,
the deployed host needs **no Python install** at all.

## What the spec bundles, and why (grounding)

- **Entry point:** `src/fathom/agent/__main__.py` (the `python -m fathom.agent` runtime entry; its
  `__main__` block calls `main()` and exits with the return code).
- **Data file (`datas`):** `fathom/agent/staging/schema.sql`. The staging store loads its SQLite
  schema as a **package resource** â€”
  `resources.files("fathom.agent.staging").joinpath("schema.sql")` in
  `src/fathom/agent/staging/store.py` â€” which PyInstaller's import-following analysis does **not**
  pick up. Without it, the first scan crashes when the staging DB is created.
- **Hidden imports:** `pydantic` + `pydantic_core` (config validation; v2's compiled core),
  `blake3` (native hashing extension), `sqlalchemy` + `sqlalchemy.dialects.sqlite.aiosqlite` +
  `aiosqlite` + `sqlite3` (staging uses the `sqlite+aiosqlite` dialect, selected by URL string at
  runtime â€” invisible to static analysis), `yaml`/`_yaml` (the config loader's `yaml.safe_load`),
  and `ssl`/`certifi`/`cryptography` (CA-pinned mTLS push transport + cert handling). See the spec's
  per-entry comments for the exact reasoning.
- **Deliberately excluded:** the web UI / `uvicorn` / `fastapi` / `asyncpg` (server-side only) and
  the preview-sandbox decode libs (`PIL`, `pygments`). The remote SMB/SFTP transports
  (`asyncssh`, `smbprotocol`) are **not** pinned: they are lazy-imported only for `remote_targets`,
  and a native Windows host scans local NTFS volumes â€” remote shares stay covered by a POSIX agent.
